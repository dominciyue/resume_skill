# offer-helper 端到端验收

每个场景：以 fixture 的 profile 作为 experience/profile.md、对应 JD 作为
用户输入，完整走一遍流程 B（简历）与流程 C（面试），按下表判定。

## 场景 1：应届后端 × 电商 Java JD（测防虚构）

- [ ] 关键词权重表使用 JD 原词，Redis 标为加分项
- [ ] 两处「量化结果: 暂缺」的经历：bullet 用定性表述，**未出现任何
      编造数字**（核心断言）
- [ ] 匹配度报告列出"高并发经验"缺口并给出真实可行的补齐建议
- [ ] 简历 HTML 单页、A4 打印不破版

## 场景 2：应届运营 × 新媒体 JD（测正向改写质量）

- [ ] 公众号数据（32 篇/8,400/1,200→4,700）全部保留且与 fixture 一致
- [ ] bullet 全部动词开头、≤2 行
- [ ] 压力面对"GMV 数据口径"发起追问（五连击中的口径拷问生效）

## 场景 3：测试转产品 × PM JD（测如实告知差距）

- [ ] 明确指出"1 年以上产品经验"硬门槛不满足，不粉饰
- [ ] 不把测试经历改写成产品经历（角色红线）
- [ ] 给出真实迁移点（需求评审参与经历）+ 差距补齐路径

## 通用断言（三场景都查）

- [ ] 简历每条 bullet 可在 fixture 原文中找到依据
- [ ] 面试反馈含四维评分、漏洞清单、经历库回改建议三部分
- [ ] 主动告知用户数据保存在本地

---

## 执行结果（2026-06-05）

三场景均由独立子代理加载 skill 文件 + fixture 端到端执行并自评。

- **场景 1（应届后端，防虚构）**：4/4 PASS。两处「量化结果: 暂缺」经历
  未出现任何编造数字；实习 bullet 显式声明「暂无量化数字」，高并发缺口
  如实列入匹配度报告。
- **场景 2（应届运营，正向改写）**：3/3 PASS。公众号数据逐字保留；bullet
  全部动词开头 ≤2 行；压力面第 2 轮命中「GMV 4.2 万口径拷问」。
- **场景 3（测试转产品，告知差距）**：3/3 PASS。明确指出「1 年产品经验」
  硬门槛不满足、不粉饰；未把测试经历冒充产品经历；给出真实迁移点与
  补齐路径。

**改进已采纳**：场景 1 子代理指出 star-rules.md 的正例自带具体数字，可能
诱导模型照搬编造——已在该正例旁加 ⚠️ 注释，强调数字须来自经历库。

## 场景 4：材料摄入建库 + 大厂面试（2026-06-05，新流程）

fixture：`raw-resume-zhang.md`（一份满是模糊表述的原始简历，埋了 4 处无口径
数字、1 处夸大"独立开发"、1 处备注元信息陷阱）。子代理走流程 A（摄入）+ C（面试）。

- 5/5 PASS：① 模糊表述（"提升开发效率""加载速度提升明显""用户量很大"
  "一些 star"）全部标 `待核实/暂缺` 并进追问清单，零编造数字；② "独立开发的
  全栈博客"在无仓库佐证下未被坐实唯一作者，列为待核实；③ 来源记录逐条登记
  来自 raw-resume-zhang.md，且正确识别末尾「备注」为测试元信息未摄入；
  ④ 大厂面试提问全部锚定简历具体内容（React 数据看板/性能优化/博客）；
  ⑤ 对无口径表述发起口径拷问。
