# 材料摄入规则

把用户提供的各种材料抽取、整合成经历库。核心约束：
**经历库每一条都要能溯源到某个材料或用户亲口确认；材料能证明什么就写
什么，证明不了的（尤其影响力数字）不许脑补。**

经历库的存放位置与运行时降级见 `SKILL.md` 的「数据存放与运行时」：有文件
写入能力时用**运行时的文件写入/编辑能力**写入 `~/offer-helper-data/profile.md`；
无文件能力的运行时，把经历库当**可导出文本**输出给用户自行保管，不谎称已保存。

## 总流程

1. 问清用户有哪些来源——**可以一次给多个仓库 + 多份文件**（例如 3 个 GitHub
   仓库 + 旧简历 + 成绩单），逐个处理。
2. 逐个来源按下面对应小节抽取，**全部汇入同一份经历库**（不是每个来源各生成
   一份）。多个仓库 = 多段项目经历并列；同一项目的多份材料相互补充。
3. 每抽一条，在「来源记录」登记它来自哪个材料。
4. 抽完后做缺口补全（只问缺失的关键字段）。
5. 按运行时规则写入/合并经历库，并把保存位置（或"请自行保存此文本"）告知用户。

> 经历库是"全集"，简历是"针对某 JD 的择优子集"：多源汇入让经历库尽量全，
> 生成简历时再按 JD 从全集里挑最匹配的 3 段，不必把所有仓库都塞进一份简历。

## 来源一：简历文件（PDF / docx / md / 图片）

- 读取文件，按教育/实习/项目/技能/获奖归类，逐段填入。
- **读不出来时**（常见于部分运行时对 .docx 的支持有限）→ 提示用户
  「请把简历另存为 PDF 后重新上传」，不要凭文件名猜内容。
- 简历是用户自己写的 → 内容可信，但**简历常缺口径**：看到"提升 50%""服务
  百万用户"这类数字，登记进库，并标注 `待核实口径`，缺口补全时问清楚。
- 简历里模糊的经历（"参与了 XX"）→ 原样登记，补全时挖具体角色和动作。

## 来源二：GitHub 仓库（URL 或本地路径）

读取方式（按可用性降级，**不要臆测仓库内容**）：

1. 有 shell + `gh`：`gh repo view <owner/repo>`、`gh api` 看 README、语言
   占比、提交历史；判断用户是 owner 还是 contributor。
2. 有 shell + git：`git clone --depth 1` 到临时目录，读 README、目录结构、
   主要语言、`git log --author=<用户>` 看其真实贡献。
3. 只有网络抓取能力（如 WebFetch / fetch / 浏览工具）：抓仓库公开页与 README。
4. 都不行（私有/离线/无网络工具）：让用户贴 README、技术栈、自己的角色。

抽取为**项目经历**，严格区分：

| 仓库能证明 | 写进库 | 仓库**不能**证明（需用户确认才写） |
|------------|--------|-------------------------------------|
| 项目存在、技术栈、用户提交了哪些代码 | 项目名、用的语言/框架、本人负责的模块 | 用户量、star 增长、性能提升、营收等影响力数字 |

- 用户只是 fork / 提了小 PR → 如实写"贡献者，提交 X"，不写成"独立开发"。
- star/fork 数是公开事实可写，但不等于"我做出了 X 影响"——影响力描述要用户
  自己提供并能解释。

## 来源三：其他文档（成绩单 / 项目报告 / 获奖证明 / 个人网站等）

- 成绩单 → GPA、排名、相关课程（用真实数字和口径）。
- 项目/实验报告 → 项目背景、本人角色、方法、结果（报告里的数据可作量化来源）。
- 获奖证明 → 奖项名称、级别、时间；获奖比例/排名若证书没写则问用户。
- 个人网站/作品集 → 按内容归入项目或作品，链接登记进可证伪细节。

## 缺口补全（摄入后，只问缺的——体验要友好，别像审讯）

对每段经历检查是否齐了五字段：背景 / 我的动作 / 量化结果 / 最难的点 /
可证伪细节。**只就缺失项提问，已有的不重复问。** 提问方式遵守：

- **批量、不阻塞**：把所有缺口**一次性列成带编号的清单**让用户挑着答，能答几条
  答几条；**一次提问不超过 3 条**，绝不一条一条连环追问。
- **给跳过出口**：明确告诉用户"想不起来的直接说『跳过』或『不知道』，我就标暂缺，
  不影响继续"——用帮回忆的口吻，不用质问口吻。
- **报进度**：开始补全前先说"还差 N 处缺口，答完就好"，给用户一个看得见的终点。
- 缺量化结果：从「规模、频次、对比、排名」四角度引导回忆；仍没有 →
  标 `量化结果: 暂缺`，**绝不代替用户编数字**。
- 角色含糊（个人还是团队）：问清本人具体做了什么。
- 标了 `待核实口径` 的数字：问清这个数是怎么算的、谁统计的。

## 无材料兜底

用户什么都没有、只想从零开始 → 退化为逐段访谈：按教育→实习→项目→技能
顺序，一次只问一段，每段挖到五字段为止。这就是缺口补全规则作用在"全部
都缺"的情形。

## 合并已有库

用户再次发来材料时：
- 同一段经历（同公司同岗位/同项目名）→ 合并，补充新细节，保留更详细的表述，
  **不删除用户已手动补充的内容**。
- 新经历 → 追加为新条目。
- 在「来源记录」追加这次的新来源。
