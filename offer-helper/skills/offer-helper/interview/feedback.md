# 面试反馈输出格式

深挖面试结束后，按以下结构输出（全部基于面试中的真实记录，不泛泛而谈）。

## 1. 各考点掌握深度（核心交付物）

逐个被深挖的考点，如实标出"钻到第几层、对照该档位期望、哪层失守"——让用户
清楚自己每个点的真实天花板、以及相对目标岗位还差哪一层。

末列给"补强动作"须满足 ACT：有主题 + 怎么补的途径 + 时长 + 验证方式；**只给
学习途径与资料方向，不写出正确答案本身**（与第 3 节"不替他背知识点"一致）；
"靠提示才答出"的层视同未真正掌握。

| 考点 | 实际钻到 | 该档位期望层级 | 失守/出错点 | 补强动作（主题·途径·时长·验证）|
|------|----------|----------------|-------------|----------------------------------|
| 如：Redis 缓存 | 第 3 层（靠提示才答出）| 校招期望第 4-5 层 | 第 4 层缓存异常防护没站住、击穿/雪崩混淆 | 主题：穿透/击穿/雪崩定义与防护；途径：官方文档+一篇权威博客通读，自己画三者对比表；约 0.5 天；验证：能不看资料复述三者触发条件与对策 |

- 某考点因时间/被引开话题只触及浅层 → 标"（仅触及第 N 层，未深入考察，非能力
  上限）"，区分"没问到"和"答不上"。
- 档位期望见 `interview/pressure.md` 前置第 3 步；阶梯层级见其第二步（1→6）。

## 2. 总评（按岗位选维度，各 1-5 分 + 一句话依据）

技术岗：技术准确性 / 技术深度（平均钻到第几层） / 表达结构 / 与简历一致性。
非技术岗：方法严谨性 / 数据敏感度 / 表达结构 / 与简历一致性。

| 维度 | 分 | 依据（引用面试中的具体回答） |
|------|----|------------------------------|
| | | |

## 3. 逐题点评

每个考点：从浅到深问了什么 → 你哪层答得好、哪层开始虚 → 那一层的正确思路
应该是什么（只点方向，基于用户真实经历，不替用户编内容也不替他背知识点）。

## 4. 简历漏洞清单（核心交付物）

「修改建议」不能只点问题，须给下面之一（具体到能照抄，又不替用户编数字）：
- **能找回数字** → 给取数动作 + **带空位模板句**，数字留空待用户填真实值。
  例："用 Redis(Cache-Aside) 重构订单查询，P99 从 ___ 降到 ___（压测 ___ 并发）"。
- **找不回** → 给定性改写模板，并标注"须经你确认属实再用"。

| 简历原文 | 暴露的问题 | 修改建议（取数动作 / 带空位模板）|
|----------|------------|-----------------------------------|
| "加 Redis 缓存…变快了很多" | 无口径、压力面问爆 | 翻压测脚本/git 找回前后 P99；填模板："P99 从 ___ 降到 ___（___ 并发）" |

## 5. 经历库回改建议

列出本次面试中用户口头补充的、但经历库里没有的真实细节
（新数字、新案例、踩坑故事），建议追加进经历库 `profile.md`
对应小节——这是经历库越用越厚的机制。

最后问用户：要不要现在就按漏洞清单更新简历和经历库？

## 6.（可选）参考方向复盘

仅当用户主动要"那这题到底怎么答"时：针对答错/失守的考点给**知识点主题方向 +
权威参考资料指引**（如"系统学一遍缓存一致性：先更库后删缓存的时序问题 + 延迟
双删 / 订阅 binlog 失效"并指向官方文档/经典资料），明确这是**补强学习、不是
面试话术，也不会替你编进简历/经历库**。默认不主动给，保持防虚构边界。
