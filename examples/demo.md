# offer-helper 是怎么工作的（原理示意）

> 注意：本文是**原理示意**，用来说明 skill 的工作方式，不是某个真实用户的
> 对话实录。真正有说服力的是**你用自己的材料现场跑出来的真实对话**——见文末
> 「如何录一条真实 demo」。

## 一、摄入：材料 → 经历库（防虚构）

把简历 / GitHub 仓库 / 文档丢给它，它抽取成结构化经历库，并**如实标注**简历
里撑不住的表述（"提升明显""用户量很大"→ 标 `待核实口径`），缺数字宁可留白，
绝不替你编。详见 [`github-intake-example.md`](github-intake-example.md)。

## 二、定制简历：针对 JD，只重组真实经历

分析 JD 关键词 → 选最匹配的经历 → STAR 量化改写 → 自检每条都能溯源 → 输出
A4 简历 + 匹配度报告（命中什么、缺什么、怎么真实补齐）。

## 三、大厂面试：抓技术点，由浅及深钻到你的边界

这是 skill 的灵魂，也是和"通用 chatbot 陪聊"最不一样的地方。它**不问"你最大
的缺点"这类套路题**，而是像真大厂面试官一样，揪住你简历里的一个技术点，
一层层往深了问，直到探到你答不上来的那层。

以"简历写了用 Redis 优化订单查询"为例，提问阶梯长这样（**示意**）：

| 层 | 面试官会问 | 在考什么 |
|----|------------|----------|
| 1 是什么 | 你用 Redis 缓存了哪些字段？怎么接入的？ | 是不是真用过 |
| 2 为什么 | 为什么不用本地缓存？多一层网络开销值吗？ | 选型判断 |
| 3 原理 | 订单状态变了，缓存怎么更新？先删缓存还是先更库？ | 一致性理解 |
| 4 边界 | 删缓存失败会怎样？穿透/击穿/雪崩防了哪些？ | 异常处理 |
| 5 口径 | QPS 多少？命中率多少？这俩数怎么统计的？ | 数字验真 |
| 6 极限 | 爆款商品热点 key 打满单分片怎么办？ | 能力天花板 |

钻到你开始"这个我们当时没考虑过……"，它就停下、记下"在第 N 层失守"，换下
一个技术点重新由浅及深。

结束后给一张**「各考点掌握深度」表**：每个技术点钻到第几层、哪层开始虚、
面试前该补哪一层——这正是面试前最该知道的东西。

## 如何录一条真实 demo（用于社媒视频）

1. 用 Claude Code 装好 skill。
2. 把你**真实的 GitHub 仓库 + 旧简历**喂给它，生成一份真简历。
3. 让它模拟面试，**你真实作答**（答不上来就如实说，效果更真）。
4. 录屏全程；剪辑时突出两个爽点：① 简历里被它揪出的"水分"；② 技术点被
   一层层问到你边界的瞬间。
5. 这条真实对话比任何编出来的脚本都有说服力。
