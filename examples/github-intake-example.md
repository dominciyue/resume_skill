# 示例：从一个 GitHub 仓库自动建立项目经历

这是 offer-helper「发个仓库就能建库」的真实走查。展示它**能从仓库证明什么、
不能脑补什么**（防虚构的关键）。示例用一个真实公开仓库
`dominciyue/resume_skill`（本 skill 自己的仓库），你可以自行核对。

---

**你**：我没空写项目经历，这是我的 GitHub 项目，帮我提取成经历库的一条：
`https://github.com/dominciyue/resume_skill`

**offer-helper**：（在有 shell 的环境）我会 `git clone --depth 1` 读它的
README、目录结构、主要语言，并用 `git log --author=你` 看你实际提交了什么。
读完得到以下**可证实**的信息：

> 仓库能证明的（直接写进经历库）：
> - 项目存在：一个中文求职 Agent Skill
> - 技术构成：Markdown 指令 + 1 个 HTML 简历模板（无后端、无网络代码）
> - 结构：SKILL.md 入口 + resume/ interview/ experience/ 三个模块 + tests/
> - 你的角色：从 commit 历史看你是 owner 且主要提交者

抽成经历库里的一条 **项目经历** 草稿：

```markdown
### offer-helper 求职 Agent Skill（个人项目 / owner，2026.06）
- 背景：中文求职场景缺乏 Agent Skill，自建一个覆盖"建库→简历→面试"的工具
- 我的动作：设计 SKILL.md 入口路由与三大模块；编写 STAR 改写、防虚构、
  大厂面试追问等规则；用纯 Markdown + HTML 模板实现，无外部依赖
- 量化结果: 暂缺（star/被使用量需你自己提供并能解释，仓库本身不证明影响力）
- 最难的点: 暂缺（待你补：设计中最棘手的取舍是什么？）
- 可证伪细节：仓库 github.com/dominciyue/resume_skill；语言 Markdown/HTML；
  防虚构红线、4 套端到端验收场景可在 tests/ 核对
```

我**不会**替你写的（仓库证明不了，硬写会在面试穿帮）：
- ❌「获得 XX star / 服务 XX 用户」——除非你提供真实数字并说明口径
- ❌「独立完成全部架构设计」——如果 commit 里有其他贡献者，按实际比例写
- ❌ 任何性能/效率提升的百分比

**需要你补 2 个缺口**（只问缺的）：
1. 这个项目有没有可量化的成果？（被多少人用 / star 数 / 收录进哪些合集）
2. 做的过程中最难的一个技术或设计取舍是什么？（面试常被追问）

---

## 这个示例说明了什么

- **省事**：你不用回忆和手写项目经历，仓库里的事实自动变成结构化草稿。
- **可信**：仓库是"可证伪细节"的天然来源——面试官点开就能看，所以基于它写
  的简历**经得起追问**。
- **防虚构**：仓库证明"做了什么、用了什么、谁做的"，但**证明不了影响力**；
  影响力数字一律留给你本人提供，这正是 offer-helper 和"顺嘴帮你吹"的
  通用 chatbot 的区别。
